# FitConnect - Project Instructions

## Overview
FitConnect is a fitness platform connecting personal trainers (professors) and students (alunos). Built with modern web tech, deployed at fitconnect.abacusai.app.

## Key Features Implemented
1. **Dynamic plan limits** - Professor dashboard and alunos-manager use dynamic maxStudents from stats API (not hardcoded)
2. **Invite flow** - Professor invites student by email → PendingInvite record created → Email sent with `/signup?type=aluno&email=xxx` link → On signup, auto-links via PendingInvite table
3. **Professor invite code** - 6-char alphanumeric code (auto-generated), shown in alunos page with copy button. Students can enter code during signup to auto-link.
4. **Pending invites display** - Alunos page shows pending invites (aguardando cadastro) with cancel option
5. **Student personal workouts** - Students can create their own workouts (isPersonal=true, professorId=null). Aluno treinos page has tabs: "Do Professor" / "Pessoais"
6. **Search/filters** - All list pages (alunos, treinos professor, treinos aluno) have search bars
7. **Workout timer** - ExecutarTreino has elapsed timer and cancel button
8. **Mobile UX** - min-h-[44px] on buttons/inputs for touch targets

## Schema Notes
- Professor.inviteCode: String? @unique - 6-char code
- PendingInvite model: email + professorId (unique combo)
- AssignedWorkout.professorId: String? (nullable for personal workouts)
- AssignedWorkout.isPersonal: Boolean @default(false)

## API Structure
- GET /api/professor/students → { students: [...], pendingInvites: [...] }
- GET /api/professor/invite-code → { inviteCode: "ABC123" }
- DELETE /api/professor/pending-invites/[id]
- POST /api/aluno/workouts → Create personal workout

## Multiple Media (implemented)
- Exercises support up to 3 media files (images/videos)
- mediaFiles Json field added to WorkoutTemplateExercise and AssignedWorkoutExercise
- Backward compatible: legacy mediaUrl/mediaType/mediaPath still read from DB
- Form shows grid of uploaded thumbnails with add button (max 3)
- All APIs (create, update, duplicate, assign) save mediaFiles array

## Google SSO (implemented)
- PrismaAdapter + GoogleProvider added to auth config
- Account and VerificationToken models added to schema
- User.passwordHash now nullable (String?) for Google SSO users
- User.userType now nullable (String?) — null means pending onboarding
- Onboarding page (/onboarding) for new Google users to choose professor/aluno
- API /api/auth/onboarding creates Professor or Student record
- Middleware redirects users without userType to /onboarding
- Login and Signup forms have "Entrar com Google" / "Cadastrar com Google" buttons
- GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET use placeholders — user must provide real credentials
- Cookies configured for state and pkceCodeVerifier (sameSite: lax)
- JWT callback refreshes userType from DB when missing
